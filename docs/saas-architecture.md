# SaaS 아키텍처 — 웹 서비스 + 로컬 게시 도우미

> 결정 배경: "아무나 가입하는 웹서비스"로 내보내고 싶다. 단, 네이버 게시는 클라우드에서 못 한다(아래 §제약).
> 결론: **UI·데이터는 클라우드(Vercel + Supabase), 네이버 게시 한 칸만 유저 컴의 로컬 도우미.**

## 핵심 제약 — 왜 게시만 로컬인가

네이버는 공식 글쓰기 API가 없어서 **에디터 자동화**([editor.py](../src/autoblog/publish/editor.py))가 유일한 길이다. 그 자동화는 진짜 브라우저 세션이 *어딘가*에서 돌아야 하는데, 그 "어딘가"는 물리적으로 둘 중 하나다:

- **클라우드(데이터센터 IP)** → 집에서 만든 세션을 데이터센터에서 쓰면 네이버가 "세션 탈취"로 보고 **계정 잠금/정지**. Vercel이든 VPS든 데이터센터 IP인 건 동일 → 못 씀.
- **유저 컴(본인 IP)** → 안전. ← 채택.

추가로 클라우드 게시는 (a) 유저 네이버 세션을 내가 보관하는 책임 폭탄, (b) 자동 글쓰기 = 약관 위반이 따라온다. 그래서 **게시 단계만** 유저 컴에서 돈다.

두 로그인은 완전 별개임을 명심:
- **서비스 로그인** = 내 앱에 "이 사람이 누구인지". Supabase Auth. 안전, 마음대로.
- **네이버 세션** = 실제 게시 권한. 서비스 로그인으로 안 생김. 로컬 도우미가 보유.

## 컴포넌트

```
[Vercel]        웹 UI (Next.js) — 가입·글쓰기·메모·수집·사진 배치·서식·미리보기·초안(LLM)
   │  로그인: Supabase Auth (이메일/소셜 — 서비스용. 네이버 아님)
   ▼
[Supabase]      Postgres(계정·본문·게시 작업 큐) + Storage(썸네일) + Realtime
   │  ← 여기까지 순수 클라우드. 설치 0.
   ▼ (작업 큐 폴링/구독, 아웃바운드 연결만)
[로컬 도우미]    트레이/메뉴바 앱 — 지금 autoblog 엔진을 PyInstaller로 포장
                 = 네이버 로그인(1회) + 로컬 사진 읽기 + Chromium으로 실제 게시
```

## 데이터 흐름 (게시)

1. 유저가 웹에서 글 작성 + 사진 배치 → 본문/레이아웃을 Supabase에 저장, 상태 `대기`.
2. 로컬 도우미(같은 계정 로그인)가 Supabase 작업 큐를 구독/폴링 → `대기` 작업 발견.
3. 도우미가 본문/레이아웃을 받고, **로컬 디스크에서 원본 사진을 직접 읽어** Chromium으로 네이버 게시.
4. 완료 → Supabase 상태 `완료` → 웹 UI 실시간 반영.

도우미는 **아웃바운드 연결만** 한다(방화벽·포트 개방 불필요). 웹이 localhost를 직접 부르지 않음 → CORS/mixed-content 회피.

## 이미지 정책 — 종류별로 저장 위치가 다름

블로그에 들어가는 이미지는 3종류고, 내 DB/스토리지에 실제로 쌓이는 건 거의 없다:

| 종류 | 내 클라우드에 | 비고 |
|---|---|---|
| **유저 본인 사진/영상** | 원본 ❌ 로컬 / 썸네일만 ✅ Storage | 글마다 생기는 진짜 유저 콘텐츠 |
| **스티커** | **이미지 0** / `pack:index` 참조만 | 네이버 자산. 도우미가 좌표로 골라 네이버가 렌더 |
| **구분선·인용구·지도** | **이미지 0** | 마커뿐, 네이버 네이티브 |

- **유저 사진**: 원본(한 글 수십 장, [data/uploads](../data/uploads))은 **로컬에 머문다**. 웹 배치/미리보기용 **가벼운 썸네일만** Storage에. 실제 게시는 도우미가 로컬 원본으로.
- **스티커**: ([stickers.py](../src/autoblog/publish/stickers.py)) 안정 식별자가 `(pack, index)`라 글엔 `clip_001:8` 같은 텍스트만 남는다. 이미지 바이트 저장 없음. 카탈로그 **메타데이터**(태그·즐겨찾기·협찬지정)는 작아서 `sticker_catalog` 테이블에 둠.
  - **썸네일 = 네이버 CDN 핫링크 (검증 완료, 저장 0).** CDN(`storep-phinf.pstatic.net`)은 외부 referer를 403으로 막지만 **referer 없으면 200**. 그래서 웹에서 `<img referrerpolicy="no-referrer">`(또는 페이지 `<meta name=referrer content=no-referrer>`)로 띄우면 내 스토리지에 0장 저장하고 그대로 렌더됨. URL은 pack+index로 결정 → `https://storep-phinf.pstatic.net/{pack}/original_{index+1}.png?type=m480_480` (OGQ 개별). 기본팩(clip/cafe/motion)은 개별 이미지 없이 스프라이트(`original_preview.png`)뿐이라 스프라이트 핫링크 후 CSS로 index 칸 크롭.
  - **폴백 B**: 네이버가 no-referer까지 막으면 수백 장을 Storage에 1회 업로드(글 활동과 무관, 바운드).

## 로컬 도우미 = 유저가 까는 것

- 윈도우 `.exe` 설치본 / 맥 `.dmg`. 설치 후 **트레이/메뉴바 아이콘**으로 백그라운드 상주(드롭박스 도우미 느낌).
- 내용물: 게시 엔진(작음) + **번들 Chromium(~150MB, 용량 대부분)** + 클라우드 연결 코드.
- 지금 `autoblog ui`/엔진을 새로 만드는 게 아니라 **PyInstaller로 다시 포장**하는 것.

### 빌드 제약 (중요)

- **PyInstaller는 크로스컴파일 불가.** 윈도우 `.exe`는 윈도우에서, 맥 `.app`은 맥에서 빌드해야 함.
- 개발 환경이 맥(darwin)이므로 윈도우 빌드는 **GitHub Actions(windows-latest)** 또는 윈도우 머신/VM 필요. → 빌드 환경 결정 미정(보류).
- **Playwright Chromium 번들링**이 패키징 최대 고비: `PLAYWRIGHT_BROWSERS_PATH`로 번들 내 Chromium을 가리켜야 함. (선택지: 용량 줄이려면 시스템 크롬 재사용 검토 — `channel="chrome"`.)
- 코드서명 미적용 시 윈도우 SmartScreen / 맥 Gatekeeper 경고(후순위).

## 리팩토링 선결 과제

- **데이터 경로 분리**: 지금 [data/](../data/)가 repo 안. 도우미 배포 시 유저 홈(`%APPDATA%` / `~/Library/Application Support`)으로 분리 필요.
- **엔진을 헤드리스 잡 실행기로**: 현재 `autoblog ui`는 로컬 웹 UI. 도우미 모드는 UI 없이 "큐 구독 → 게시" 루프.
- **세션 보관**: 네이버 세션(`naver_state.json`)은 **로컬에만**. 절대 클라우드로 안 보냄.
